Part 1 & 2: OpenMP

Shared Memory Parallelization (115 Measurements)

Design: Cache-Conscious Implementation

Problem: Matrix multiply (C = A × B) needs fast B access

Solution: Transpose B beforehand

Original: B[k,j] read column-wise (slow, cache misses)
After transpose: B'[j,k] read row-wise (fast, cache hits)
Result: Enables SIMD vectorization by compiler

Key Finding: Size Matters!

Small matrices (50×50): Threading HURTS
Thread startup and synchronization overhead > computational work
With 2 threads: 0.26× speedup (26% SLOWER!)
Large matrices (1000+×1000+): Threading HELPS
Parallelism amortizes overhead
Speedup reaches 6.1× with 16 threads
Lesson: Always profile before parallelizing!

Part 3: MPI

Distributed Memory Computing (4 Processes)

Architecture: Column-Wise Partitioning

Each of 4 processes gets N/4 columns of the result matrix

  • Compute: O(N³/P) per process (highly parallelizable)
  • Communicate: One AllReduce at end (tree-structured)
  • Cost: ~log(P) communication rounds
Result: Clean linear speedup, minimal communication overhead

Part 4: GPU

OpenCL on NVIDIA A10 Accelerator

Two Kernel Implementations

Naive Kernel
for (int j = 0; j < N; j++) {
  for (int k = 0; k < N; k++) {
    C[i,j] += A[i,k] * B[k,j];  // Random memory access
  }
}

❌ Low cache efficiency, high memory bandwidth

Optimized Kernel (Tiling)
__local float tA[32][32], tB[32][32];
// Load tiles, synchronize, compute
// Reuse data 32× before reload
// Local memory: 1000 GB/s vs Global: 400 GB/s

✓ 32× data reuse, better bandwidth

Theoretical Analysis

Scaling Laws & Amdahl's Law

Scaling Laws Analysis

Amdahl's Law • Predicted vs actual • Strong scaling efficiency • Memory bandwidth

What Amdahl's Law Tells Us

Formula: S(P) = 1 / [(1-p) + p/P]
where p = parallelizable fraction, P = processors
Our Data: Estimated serial fraction ≈ 5-10%
This explains the speedup plateau at 16 threads
With 100% parallelizable code: would expect 16× speedup
With 5% serial: max speedup ≈ 20×, but we see 6.1×

The bottleneck is memory bandwidth, not parallelism!

Key Lessons

What We Learned

  1. Measurement is Essential
    Theory says parallelism should help—surprising to see small slowdown
    → Reveals that operation is memory-bound
  2. Problem Size Matters
    Small matrices: overhead dominates
    Large matrices: parallelism amortizes overhead
  3. Different Tools for Different Jobs
    OpenMP: Fast, simple, limited to 1 machine
    MPI: Scales across cluster, more complex
    GPU: Massive parallelism, I/O overhead
  4. Theory ≠ Practice
    Textbooks are guides, not guarantees
    Real hardware has caches, bandwidth limits, overhead
