Goal

Implement matrix multiplication C = A × B on all four paradigms and compare performance

Part 1 & 2: OpenMP Parallelism

Sequential + Parallel Implementation

Key Optimization: Transpose matrix B

  • ✓ Improves cache locality - read both matrices sequentially
  • ✓ Enables SIMD auto-vectorization by compiler
  • ✓ Implementation: #pragma omp parallel for

Parallelization Strategy

Distribute outer loop iterations across threads:

#pragma omp parallel for collapse(2)
for (int i = 0; i < n; i++)
  for (int j = 0; j < n; j++)
    C[i][j] = dot product of A[i] and B_transposed[j]

Why Limited Speedup?

Amdahl's Law: Speedup = 1 / (f_s + f_p/P)

  • Memory bandwidth saturation limits speedup
  • CPU can't feed data to cores fast enough
  • Thread overhead outweighs benefit for small matrices
Key Insight: This is memory-bound, not compute-bound. Adding more threads won't help if memory bandwidth is the bottleneck.

Part 3: MPI Distributed Computing

Column-wise Matrix Partitioning

Each MPI process handles a subset of result columns:

  • Process 0: Columns 0 to n/p-1
  • Process 1: Columns n/p to 2n/p-1
  • ...

Algorithm

  1. Broadcast matrices A and B to all processes
  2. Each process computes its portion locally
  3. MPI_Allreduce gathers all results
  4. Root process has complete result

Expected Benefits

  • Computation runs in parallel on N processes
  • Communication overhead: O(n²) gather (small relative to computation)

Why MPI Outperforms Sequential

  • ✓ 4 processes compute simultaneously
  • ✓ Each handles 1/4 of the work
  • ✓ Allreduce is extremely fast for single gather
  • ✓ 1.7× speedup vs sequential (strong scaling)
Key Insight: Column partitioning eliminates communication bottleneck. Communication is negligible!

Part 4: GPU Computing with OpenCL

Two Kernel Implementations

Naive Kernel (Global Memory Only)

__kernel void matmul_naive(__global float* A, __global float* B,
                        __global float* C, int n) {
  int i = get_global_id(0), j = get_global_id(1);
  float sum = 0.0f;
  for(int k = 0; k < n; k++)
    sum += A[i*n + k] * B[k*n + j];
  C[i*n + j] = sum;
}

Optimized Kernel (Local Memory Tiling)

Load tiles into fast local memory (16×16 blocks)

  • Reduces global memory bandwidth by ~16×
  • Work-group synchronization for data coherency
  • Expected speedup: 2-4× vs naive

GPU Results & Analysis

Kernel Optimization Impact

Kernel Memory Model Est. Performance Speedup
Naive Global only ~Y GFLOPs 1.0×
Optimized Local + Global ~W GFLOPs ~2-4×

Why Local Memory Matters

  • Global Memory: 100-200 GB/s (shared across all cores)
  • Local Memory: 1-10 TB/s (on-chip, per work-group)
  • Result: 50-100× bandwidth improvement!

GPU Data Transfer Analysis

The Hidden Cost

  • PCIe Bandwidth: ~16 GB/s (PCIe 4.0)
  • 1000×1000 matrix: 24 MB data (3 matrices × 8 bytes)
  • Transfer time: ~1.5 ms (much faster than computation!)
  • BUT: Only amortized over repeated operations

When GPU Makes Sense

GPU best when: Computation time >> Transfer time

  • ✓ Large matrices (> 2000×2000)
  • ✓ Multiple operations on same data
  • ✓ Streaming data (video processing)
  • ✗ One-time small operations

Lessons Learned

Key Takeaways

  1. Cache Locality Matters: Simple transpose improved performance significantly
  2. Measure Communication: MPI showed communication is negligible with smart partitioning
  3. Memory is the Bottleneck: OpenMP speedup limited by bandwidth, not computation
  4. Different Problems, Different Solutions: No one paradigm beats all
  5. Amdahl's Law is Real: Speedup limited by sequential fraction

Why These Design Choices?

Sequential: Matrix Transpose

  • Sequential access to both A and B improves cache hits
  • Enables compiler SIMD optimization
  • Small overhead (O(n²) one-time cost)

MPI: Column Partitioning

  • Each process owns full rows (cache-friendly access)
  • Only needs single MPI_Allreduce (minimal communication)
  • Easy to implement and understand

GPU: Local Memory Tiling

  • Dramatic bandwidth improvement (50-100×)
  • Work-group synchronization ensures correctness
  • Standard GPU optimization technique

Common Questions

Q: Why not use GPU for everything?

A: Data transfer overhead, not all problems are data-parallel, programming complexity

Q: How would this scale to very large matrices?

A: GPU memory limits (~10GB), would need MPI+GPU hybrid approach

Q: Why did OpenMP not scale better?

A: Memory-bound operation - CPU bandwidth is shared across cores

Q: Which paradigm is "fastest"?

A: Depends on problem size and platform. GPU best for large matrices, MPI best for multi-node scalability
